# BScrap — Техническое задание

## 1. Общее описание

**BScrap** — универсальная платформа для приёма, обработки и индексации контента из различных источников. Ядро системы — FastAPI-сервер, принимающий текст и файлы от внешних программ-коннекторов (email, Telegram, YouTube и др.), преобразующий любой контент в текст, генерирующий эмбеддинги и сохраняющий результат в PostgreSQL (pgvector).

---

## 2. Архитектура

```
Коннекторы (внешние программы)
        │
        ▼
   FastAPI сервер ──► Очередь задач (SQLite) ──► Worker(ы)
                                                     │
                                    ┌────────────────┼────────────────┐
                                    ▼                ▼                ▼
                             Извлечение        Генерация        Запись в БД
                             текста из         эмбеддингов      (PostgreSQL
                             контента          (Ollama)          + pgvector)
```

### Компоненты:

- **FastAPI** — HTTP API для приёма контента
- **Очередь задач** — SQLite-based, персистентная, FIFO
- **Worker pool** — N процессов (настраивается), обработка задач
- **Content processors** — модули извлечения текста из разных форматов (базовые + композитные)
- **Embedding engine** — генерация эмбеддингов через Ollama
- **PostgreSQL + pgvector** — хранение данных и векторов

> **Примечание**: Ollama обрабатывает запросы последовательно (внутренняя очередь). При `num_workers > 1` параллелизм ускоряет только подготовительные операции (ffmpeg, извлечение текста, конвертация файлов), но **не** этапы vision/embedding — запросы к Ollama от разных воркеров будут выстраиваться в очередь.

## 3. Функциональные требования

### 3.1 FastAPI сервер

#### Эндпоинты:

**`POST /process`**
- Принимает: multipart/form-data
  - `text` (string, optional) — текстовый контент
  - `files` (list[UploadFile], optional) — файлы любых поддерживаемых форматов
  - `source_service` (string, required) — название сервиса-источника (telegram, youtube, email, и т.д.)
  - `source_url` (string, optional) — ссылка на первоисточник
  - `metadata` (JSON string, optional) — произвольные метаданные, специфичные для сервиса
- Лимит размера загружаемых файлов: настраивается в config.yaml (`max_upload_size_mb`, дефолт 2048 МБ). При превышении — HTTP 413 Request Entity Too Large
- Проверка дубликатов: если запись с таким `source_service` + `source_url` уже существует — HTTP 409 Conflict с телом `{ "content_record_id": "uuid" }`
- Возвращает: `{ "task_id": "uuid" }` (HTTP 202 Accepted)
- Аутентификация: не требуется

**`GET /task/{task_id}`**
- Возвращает:
  - `status`: `pending` | `processing` | `completed` | `failed`
  - `result` (при completed):
    - `generated_text` — извлечённый/сгенерированный текст
    - `vector_record_ids` — список ID записей в векторной БД
  - `error` (при failed): описание ошибки

**`GET /health`**
- Проверяет:
  - Подключение к PostgreSQL
  - Доступность Ollama + наличие сконфигурированных моделей (embedding_model, vision_model)
  - Доступность CUDA (если `whisper.device: cuda`)
  - Наличие `ffmpeg` в PATH
  - Наличие `libreoffice` в PATH
- Возвращает:
  - HTTP 200 — все проверки пройдены, тело `{ "status": "healthy", "checks": { ... } }`
  - HTTP 503 — одна или более проверок не пройдены, тело `{ "status": "unhealthy", "checks": { ... } }`
- **При старте системы** вызывается автоматически. Если healthcheck не пройден — система **не запускается** и выводит в лог детали ошибки.

### 3.2 Очередь задач

- Хранение: SQLite (файл, путь настраивается в config.yaml)
- Поведение:
  - FastAPI добавляет задачу в очередь
  - Worker забирает задачу → задача помечается как `processing`
  - При успешной обработке — задача удаляется из очереди
  - При сбое/крэше воркера — задача записывается в crash log (см. п. 3.7), статус в `task_log` устанавливается в `failed`

### 3.3 Обработка контента

#### Архитектура процессоров

Система использует два уровня процессоров:

**Базовые процессоры** — атомарные единицы обработки:

| Процессор | Вход | Выход |
|-----------|------|-------|
| `TextProcessor` | plain text, HTML (BeautifulSoup для стрипания тегов/скриптов/стилей), markdown | Чистый текст |
| `ImageProcessor` | jpg, png, gif, bmp, webp, tiff, svg | Текстовое описание (vision) + OCR |
| `AudioProcessor` | mp3, wav, ogg, flac, m4a | Транскрипция (faster-whisper) |

**Композитные процессоры** — собираются из базовых:

| Процессор | Логика |
|-----------|--------|
| `VideoProcessor` | ffmpeg → ключевые кадры → `ImageProcessor` + аудиодорожка → `AudioProcessor` |
| `PdfProcessor` | Рендер каждой страницы в изображение → `ImageProcessor` |
| `OfficeProcessor (docx)` | Извлечение текста → `TextProcessor` + встроенные изображения → `ImageProcessor` |
| `OfficeProcessor (xlsx/ods)` | Каждый лист → текст (markdown/CSV таблица) → `TextProcessor` |
| `OfficeProcessor (pptx/odp)` | Рендер каждого слайда в изображение → `ImageProcessor` |
| `LibreOfficeConverter (odt, ods, odp)` | Конвертация в MS Office формат через libreoffice CLI → далее соответствующий OfficeProcessor |

Все процессоры (базовые и композитные) наследуются от `BaseProcessor` — абстрактного класса с методом `process(file_path: Path) -> str`. Композитные процессоры принимают экземпляры базовых через конструктор (dependency injection).

#### Детали обработки:

**Изображения (`ImageProcessor`):**
- Передаются в Ollama vision-модель
- Результат: текстовое описание содержимого + распознанный текст (OCR)
- Если OCR ничего не распознал — возвращается пустая строка для OCR-части

**Видео (`VideoProcessor`):**
- ffmpeg извлекает ключевые кадры с заданной частотой (`video.frame_extraction_fps`)
- Лимит: `video.max_frames` (дефолт 1000). Если кадров при заданной частоте больше лимита — частота автоматически уменьшается, чтобы равномерно покрыть всё видео. В лог записывается WARNING с указанием исходной и итоговой частоты
- Каждый кадр → `ImageProcessor` → описание + OCR
- Аудиодорожка → `AudioProcessor` → транскрипция
- Если аудиодорожки нет — транскрипция = пустая строка
- Результат собирается по шаблону:

```
Видео. Транскрипция: "<транскрипция с таймкодами>"
Описания кадров: "<описания кадров с таймкодами>"
OCR: "<OCR ключевых кадров с таймкодами>"
```

**PDF (`PdfProcessor`):**
- Каждая страница рендерится в изображение → `ImageProcessor` (описание + OCR)
- Лимит: `pdf.max_pages` (дефолт 1000). Если страниц больше лимита — страницы равномерно выбираются из всего документа. В лог записывается WARNING. 
- Извлечение встроенных изображений → `ImageProcessor`

**Excel (xlsx/ods):**
- Каждый лист → CSV таблица → `TextProcessor`

**PowerPoint (pptx/odp):**
- Каждый слайд рендерится в изображение → `ImageProcessor`

**Word (docx):**
- Извлечение текста → `TextProcessor`
- Встроенные изображения → `ImageProcessor`

### 3.4 Вложения / файлы

- Входящие файлы сохраняются во временную папку (temp, путь в config.yaml)
- В БД путь к файлам **не записывается** (кроме таблицы `content_files`, если файл сохранён)
- После обработки: оригинальные файлы **удаляются или сохраняются** (настраивается в config.yaml: `keep_original_files: true/false`)
- Если `keep_original_files: true` — файлы перемещаются в папку хранения (путь в config.yaml), но только если размер файла не превышает `max_file_size_mb` (лимит на долгосрочное хранение)

### 3.5 Эмбеддинги

- Генерация: Ollama (модель настраивается в config.yaml)
- Chunking: семантический, библиотека `langchain_text_splitters` (RecursiveCharacterTextSplitter)
  - Размер чанка: настраивается в config.yaml
  - Overlap: настраивается в config.yaml
- Каждый чанк → эмбеддинг → запись в PostgreSQL (pgvector)
- Связь чанков с исходной записью через foreign key

### 3.6 Дубликаты

- Проверка по `source_service` + `source_url`
- Если запись с таким источником уже существует — HTTP 409 Conflict, тело `{ "content_record_id": "uuid" }`
- Проверка выполняется до постановки задачи в очередь

### 3.7 Crash log

- При крэше воркера во время обработки задачи — данные задачи записываются в crash log
- Формат: текстовый файл с JSON-строками (одна строка = одна задача)
- Путь: настраивается в config.yaml (дефолт `./logs/crash.log`)
- Программа **только пишет** в crash log, автоматического переигрывания задач нет
- В `task_log` статус задачи устанавливается в `failed`

## 4. Модель данных (PostgreSQL + pgvector)

### Таблица `content_records`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | Уникальный идентификатор |
| source_service | VARCHAR | Название сервиса (telegram, youtube, email...) |
| source_url | TEXT (nullable) | Ссылка на первоисточник |
| original_text | TEXT | Исходный текст (если передан) |
| generated_text | TEXT | Полный извлечённый/сгенерированный текст (без ограничения размера) |
| metadata | JSONB | Произвольные метаданные сервиса |
| created_at | TIMESTAMP | Время создания записи |
| processed_at | TIMESTAMP | Время завершения обработки |

**Constraints:**
- UNIQUE (`source_service`, `source_url`) — для проверки дубликатов

### Таблица `content_chunks`

| Поле              | Тип       | Описание                                          |
| ----------------- | --------- | ------------------------------------------------- |
| id                | UUID (PK) | Уникальный идентификатор чанка                    |
| content_record_id | UUID (FK) | Ссылка на content_records (ON DELETE CASCADE)     |
| chunk_index       | INTEGER   | Порядковый номер чанка                            |
| chunk_text        | TEXT      | Текст чанка                                       |
| embedding         | VECTOR(N) | Вектор эмбеддинга (размерность зависит от модели) |
| created_at        | TIMESTAMP | Время создания                                    |

### Таблица `content_files`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | Уникальный идентификатор |
| content_record_id | UUID (FK) | Ссылка на content_records (ON DELETE CASCADE) |
| original_filename | VARCHAR | Имя файла |
| file_type | VARCHAR | MIME-тип |
| file_size | BIGINT | Размер в байтах |
| extracted_text | TEXT | Извлечённый из файла текст |
| storage_path | TEXT | Путь на диске (если сохранён) |
| is_stored | BOOLEAN | true — файл сохранён на диске, false — удалён |
| created_at | TIMESTAMP | Время создания |

### Таблица `task_log`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | task_id |
| status | VARCHAR | pending / processing / completed / failed |
| source_service | VARCHAR | Сервис-источник |
| source_url | TEXT (nullable) | Ссылка |
| content_record_id | UUID (FK, nullable) | Ссылка на результат |
| error_message | TEXT | Ошибка (если failed) |
| created_at | TIMESTAMP | Время создания задачи |
| started_at | TIMESTAMP | Время начала обработки |
| completed_at | TIMESTAMP | Время завершения |

### 5.4 Whisper: режим lazy_load

При `whisper.lazy_load: true` транскрипция выполняется через отдельный скрипт `bscrap/processors/whisper_worker.py`, запускаемый как subprocess:

- Вызов: `python bscrap/processors/whisper_worker.py --input <путь_к_аудио> --model <модель> --device <устройство> --language <язык|auto>`
- Скрипт загружает модель, выполняет транскрипцию, выводит результат в stdout как JSON и завершается
- Формат stdout: `{ "text": "<транскрипция>", "language": "<определённый_язык>" }`
- Это позволяет освободить VRAM после каждой транскрипции

При `whisper.lazy_load: false` модель загружается один раз при старте воркера и остаётся в памяти.

### 5.5 Логирование

- **Консоль**: стандартный `logging`, human-readable формат, уровень настраивается
- **Файл**: JSON-формат, один файл на запуск, имя = дата запуска (например `2026-04-04_14-30-00.log`)
- **Папка**: `./logs/` (настраивается в config.yaml)
- **Содержимое**: timestamp, level, module, message, task_id (если применимо), traceback (при ошибках)
- **Crash log**: отдельный файл (`crash_log_path`), формат — JSON-строки, только запись, без автоматического переигрывания

### 5.6 Добавление нового процессора

Для поддержки нового формата:
1. Определить тип: базовый (новый тип данных) или композитный (комбинация существующих базовых)
2. Создать класс-наследник `BaseProcessor` в соответствующей папке (`base_processors/` или `composite_processors/`)
3. Реализовать метод `process(file_path: Path) -> str`
4. Для композитных — принять нужные базовые процессоры через конструктор
5. Зарегистрировать в `registry.py` с привязкой к MIME-типам

### 5.7 Тестирование

- pytest для каждого модуля и класса
- Структура тестов зеркалирует структуру `bscrap/`
- Фикстуры в `conftest.py`

### 5.8 Запуск и остановка

- **Запуск**: `python main.py` → healthcheck → если все проверки пройдены → запуск сервера и воркеров
- **Остановка**: при SIGTERM/SIGINT — немедленное завершение (kill воркеров без ожидания). Незавершённые задачи откатывается до первоначального статуса(до начала обработки)

## 6. Нефункциональные требования

- **Расширяемость**: новый формат контента = новый процессор + регистрация
- **Отказоустойчивость**: очередь задач персистентная (SQLite); при крэшах — задачи логируются в crash log, статус в БД — `failed`
- **Наблюдаемость**: подробное логирование каждого этапа обработки
- **Модульность**: каждый компонент независим и тестируем отдельно; композитные процессоры собираются из базовых через DI
- **Конфигурируемость**: всё что возможно — в config.yaml, секреты — в .env
- **Явность над неявностью**: система не запускается при невалидной конфигурации (нет CUDA, нет ffmpeg и т.д.); ошибки сообщаются явно
