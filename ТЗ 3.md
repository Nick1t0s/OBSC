# BScrap — Техническое задание v2

## 1. Назначение

**BScrap** — сервер для приёма, обработки и индексации контента из внешних источников. Принимает текст и файлы по HTTP, извлекает из них текст (OCR, транскрипция, конвертация), генерирует векторные эмбеддинги и сохраняет результат в PostgreSQL + pgvector.

Поиск по эмбеддингам **не входит** в scope — данные потребляются другим сервисом напрямую из БД.

Деплой — прямой запуск на сервере (`python main.py`), без Docker.

---

## 2. Архитектура

```
Коннекторы (внешние программы)
        │
        ▼
   FastAPI сервер ──► Очередь задач (SQLite) ──► Worker (1 шт., встроенный)
                                                     │
                                    ┌────────────────┼────────────────┐
                                    ▼                ▼                ▼
                             Извлечение        Генерация        Запись в БД
                             текста из         эмбеддингов      (PostgreSQL
                             контента          (fast_ai)         + pgvector)
```

### Компоненты

| Компонент | Описание |
|-----------|----------|
| **FastAPI** | HTTP API для приёма контента |
| **Очередь задач** | SQLite, персистентная, FIFO. Хранит полный payload задачи |
| **Worker** | Один встроенный поток/процесс, поллит очередь каждые N секунд |
| **Процессоры** | Извлечение текста из разных форматов. Используют `fast_ai/` и существующие `processor/` как зависимости |
| **Эмбеддинги** | Chunking + генерация эмбеддингов через `fast_ai` (Ollama / OpenAI) |
| **PostgreSQL + pgvector** | Хранение контент-записей, файлов и эмбеддингов |

### Зависимость от существующего кода

`fast_ai/` и `processor/` используются **как зависимости** — код не копируется внутрь `bscrap/`, а импортируется напрямую. Переиспользуется:

- **`fast_ai.llm.FastLLM`** — провайдеры LLM с retry/rounds логикой
- **`fast_ai.ocr.OCR`** — OCR изображений (vision-based)
- **`fast_ai.describer.ImageDescriber`** — описание изображений
- **`fast_ai.stt.STT`** — транскрипция аудио (faster-whisper / OpenAI)
- **`processor.simple_processor.photo.PhotoProcessor`** — обработка изображений (OCR + описание по шаблону)
- **`processor.simple_processor.audio.AudioProcessor`** — обработка аудио (STT по шаблону)
- **`processor.complex_processor.pdf.PDFProcessor`** — рендер страниц PDF → PhotoProcessor
- **`processor.complex_processor.word.WordProcessor`** — DOCX → PDF → PhotoProcessor (через LibreOffice)
- **`processor.complex_processor.powerpoint.PowerPointProcessor`** — PPTX → PDF → PhotoProcessor (через LibreOffice)
- **`processor.complex_processor.xlsx.XLSXProcessor`** — XLSX → CSV по листам
- **`processor.complex_processor.ods.ODSProcessor`** — ODS → XLSX → CSV (через LibreOffice)
- **`processor.complex_processor.odt.ODTProcessor`** — ODT → PDF → PhotoProcessor (через LibreOffice)
- **`processor.complex_processor.odp.ODPProcessor`** — ODP → PDF → PhotoProcessor (через LibreOffice)
- **`processor.complex_processor.odg.ODGProcessor`** — ODG → PDF → PhotoProcessor (через LibreOffice)

Все эти модули собираются через свои `build(config_path)` + `run(source)`. BScrap **не дублирует** эту логику — он оборачивает вызовы `build()` / `run()` в свой pipeline.

## 3. API

Аутентификации нет. Rate limiting нет. Версионирования URL нет. Лимита на количество файлов в запросе нет.

### 3.1 `POST /process`

Принимает: `multipart/form-data`

| Поле | Тип | Обязательное | Описание |
|------|-----|:---:|----------|
| `text` | string | нет | Текстовый контент (plain text, HTML, markdown) |
| `files` | list[UploadFile] | нет | Файлы любых поддерживаемых форматов |
| `source_service` | string | **да** | Название сервиса-источника (`telegram`, `youtube`, `email`, ...) |
| `source_url` | string | нет | Ссылка на первоисточник |
| `metadata` | JSON string | нет | Произвольные метаданные, специфичные для сервиса |

**Лимит размера загрузки:** настраивается в `config.yaml` (`max_upload_size_mb`, дефолт 2048 МБ). При превышении — HTTP 413.

**Проверка дубликатов:** выполняется **только если `source_url` заполнен**. Если запись с таким `source_service` + `source_url` уже есть — HTTP 409 с телом `{ "content_record_id": "<uuid>" }`. Если `source_url` не передан — проверка не выполняется, запись всегда создаётся.

**Поведение:**
1. Валидация входных данных
2. Проверка дубликатов (если `source_url` заполнен)
3. Сохранение файлов во временную папку
4. Создание задачи в SQLite-очереди (полный payload: текст, пути к файлам, метаданные)
5. Создание записи в `task_log` со статусом `pending`
6. Возврат `{ "task_id": "<uuid>" }` — HTTP 202 Accepted

### 3.2 `GET /task/{task_id}`

Возвращает статус задачи.

**Ответ:**

```json
{
  "task_id": "uuid",
  "status": "pending | processing | completed | failed",
  "result": {
    "generated_text": "весь отрендеренный текст + входной текст",
    "content_record_id": "uuid",
    "chunk_ids": ["uuid", "uuid", "..."]
  },
  "error": "описание ошибки (только при failed)"
}
```

- `result` присутствует только при `status: completed`
- `error` присутствует только при `status: failed`
- `generated_text` — это склейка входного `text` и `render()`-результатов всех обработанных файлов

### 3.3 `GET /tasks`

Список задач с фильтрацией и пагинацией.

**Query-параметры:**

| Параметр | Тип | По умолчанию | Описание |
|----------|-----|:---:|----------|
| `status` | string | — | Фильтр по статусу (`pending`, `processing`, `completed`, `failed`) |
| `source_service` | string | — | Фильтр по сервису |
| `limit` | int | 50 | Количество записей на страницу (макс. 200) |
| `offset` | int | 0 | Смещение |

**Ответ:**

```json
{
  "tasks": [
    {
      "task_id": "uuid",
      "status": "completed",
      "source_service": "telegram",
      "source_url": "https://...",
      "created_at": "2026-04-19T12:00:00Z",
      "started_at": "2026-04-19T12:00:01Z",
      "completed_at": "2026-04-19T12:00:15Z"
    }
  ],
  "total": 150,
  "limit": 50,
  "offset": 0
}
```

### 3.4 `GET /health`

Проверяет готовность системы.

**Проверки:**
1. Подключение к PostgreSQL
2. Доступность Ollama + наличие сконфигурированных моделей (`embedding_model`, `vision_model`)
3. Доступность CUDA (если `whisper.device: cuda`)
4. Наличие `ffmpeg` в PATH
5. Наличие `libreoffice` (soffice) в PATH

**Ответ:**
- HTTP 200 — все проверки пройдены: `{ "status": "healthy", "checks": { ... } }`
- HTTP 503 — есть ошибки: `{ "status": "unhealthy", "checks": { ... } }`

**При старте системы** healthcheck выполняется автоматически. Если не пройден — система **не запускается**, в лог выводятся детали ошибки.

## 4. Очередь задач

### Хранение: SQLite

Файл SQLite хранит очередь задач. Путь настраивается в `config.yaml`.

### Схема таблицы `queue`

| Поле | Тип | Описание |
|------|-----|----------|
| id | TEXT (PK) | task_id (UUID) |
| payload | TEXT | JSON с полным payload задачи |
| status | TEXT | `pending` / `processing` |
| created_at | TEXT | ISO timestamp |

**Payload содержит:**

```json
{
  "task_id": "uuid",
  "text": "входной текст или null",
  "files": [
    {
      "original_filename": "photo.jpg",
      "temp_path": "/tmp/bscrap/abc123/photo.jpg",
      "mime_type": "image/jpeg",
      "size": 1234567
    }
  ],
  "source_service": "telegram",
  "source_url": "https://...",
  "metadata": {}
}
```

### Жизненный цикл задачи в очереди

1. `POST /process` → вставка в `queue` со статусом `pending`
2. Worker забирает задачу → статус в очереди меняется на `processing`
3. **Повторная проверка дубликатов** (если `source_url` заполнен) — между постановкой в очередь и началом обработки мог прийти дубликат
4. При успешной обработке → задача **удаляется** из очереди
5. При ошибке → задача записывается в crash log, **удаляется** из очереди, статус в `task_log` → `failed`

### Polling

Worker поллит очередь каждые `poll_interval_seconds` (дефолт 2 секунды). Уведомлений нет — только polling.

## 5. Worker

Worker — **один**, встроенный в процесс `main.py`. Не убивается при остановке (см. п. 10).

### Pipeline обработки одной задачи

```
1. Забрать задачу из SQLite-очереди (статус → processing)
2. Обновить task_log: статус → processing, started_at → now()
3. Повторная проверка дубликатов (если source_url заполнен)
   └─ Если дубликат найден → задача завершается, task_log → completed,
      content_record_id → существующая запись
4. Обработка текста (если есть):
   └─ TextProcessor: strip HTML/markdown → чистый текст
5. Обработка каждого файла:
   └─ По MIME-типу/расширению определить процессор → вызвать run() → получить render()
6. Склейка результата: generated_text = входной_текст + "\n\n" + render() каждого файла
7. Сохранение в PostgreSQL:
   ├─ INSERT в content_records (original_text, generated_text, metadata, ...)
   ├─ INSERT в content_files (по одной записи на каждый файл)
   └─ Chunking generated_text → эмбеддинги → INSERT в content_chunks
8. Удаление временных файлов
9. Удаление задачи из SQLite-очереди
10. Обновить task_log: статус → completed, completed_at → now()
```

### Определение процессора по файлу

Маппинг расширения → процессор (реестр в `bscrap/processors/registry.py`):

| Расширение | Процессор | Что делает |
|------------|-----------|------------|
| jpg, jpeg, png, gif, bmp, webp, tiff, svg | `PhotoProcessor` | OCR + описание (2 отдельных запроса к vision через `fast_ai`) |
| mp3, wav, ogg, flac, m4a | `AudioProcessor` | Транскрипция через `fast_ai.stt.STT` |
| mp4, avi, mkv, mov, webm | `VideoProcessor` | Ключевые кадры → `PhotoProcessor` + аудио → `AudioProcessor` |
| pdf | `PDFProcessor` | Рендер страниц → `PhotoProcessor` |
| doc, docx | `WordProcessor` | Конвертация в PDF → `PDFProcessor` |
| ppt, pptx | `PowerPointProcessor` | Конвертация в PDF → `PDFProcessor` |
| xlsx | `XLSXProcessor` | Каждый лист → CSV |
| odt | `ODTProcessor` | Конвертация в PDF → `PDFProcessor` |
| odp | `ODPProcessor` | Конвертация в PDF → `PDFProcessor` |
| ods | `ODSProcessor` | Конвертация в XLSX → CSV |
| odg | `ODGProcessor` | Конвертация в PDF → `PDFProcessor` |
| htm, html | `TextProcessor` | Strip HTML → чистый текст |
| txt, md, csv, json | `TextProcessor` | Strip markdown → чистый текст |

Неизвестное расширение → ошибка, файл пропускается, в лог — WARNING с именем файла и расширением.

## 6. Процессоры

### 6.1 TextProcessor (новый, пишется в bscrap)

Единственный процессор, который **не** существует в текущем коде и пишется с нуля.

**Вход:** plain text, HTML, markdown (строка или файл).

**Выход:** чистый текст без тегов, стилей, скриптов.

**Логика:**
- HTML → BeautifulSoup: удалить теги `<script>`, `<style>`, `<head>`, затем `get_text(separator="\n")`
- Markdown → strip markdown-разметки до plain text (библиотека на выбор, например `markdown` + BeautifulSoup, или regex-based strip)
- Plain text → возвращается как есть
- Определение формата: по расширению файла (`.html` / `.htm` → HTML, `.md` → markdown, остальное → plain text). Если передана строка без указания формата — пытаемся определить автоматически (есть ли HTML-теги), иначе plain text

**Интерфейс:**

```python
class TextProcessor:
    def process(self, text: str, *, format: str = "auto") -> str:
        """
        format: "html", "markdown", "plain", "auto"
        Возвращает чистый текст.
        """
```

### 6.2 ImageProcessor (обёртка над существующим PhotoProcessor)

Используется `processor.simple_processor.photo.PhotoProcessor` как есть.

- OCR и описание — это **2 отдельных запроса** к vision-модели через `fast_ai.ocr.OCR` и `fast_ai.describer.ImageDescriber`
- Результат: `Photo.render()` — отрендеренный текст по шаблону с переменными `$ocr`, `$description`

### 6.3 AudioProcessor (обёртка над существующим)

Используется `processor.simple_processor.audio.AudioProcessor` как есть.

- Транскрипция через `fast_ai.stt.STT` (faster-whisper / OpenAI)
- Результат: `Audio.render()` — текст транскрипции по шаблону с переменной `$text`

### 6.4 VideoProcessor (новый, пишется в bscrap)

**Вход:** видеофайл (mp4, avi, mkv, mov, webm).

**Логика:**
1. `ffmpeg` извлекает ключевые кадры с заданной частотой (`video.frame_extraction_fps`, дефолт 1 кадр/сек)
2. Если при заданном fps кадров больше `video.max_frames` (дефолт 1000) — fps автоматически уменьшается, чтобы равномерно покрыть всё видео. В лог — WARNING с исходным и итоговым fps
3. Каждый кадр → `PhotoProcessor.run()` → описание + OCR
4. `ffmpeg` извлекает аудиодорожку → `AudioProcessor.run()` → транскрипция
5. Если аудиодорожки нет — транскрипция = пустая строка

**Таймкоды:** формат `HH:MM:SS`. Вычисляются из номера кадра и fps.

**Шаблон результата:**

```
Видео. Транскрипция: "<транскрипция с таймкодами>"
Описания кадров: "<описания кадров с таймкодами>"
OCR: "<OCR ключевых кадров с таймкодами>"
```

Пример:

```
Видео. Транскрипция: "[00:00:00] Привет, сегодня мы поговорим... [00:00:15] Начнём с первого пункта..."
Описания кадров: "[00:00:00] На экране заставка канала с логотипом. [00:00:05] Ведущий за столом, на фоне белая доска."
OCR: "[00:00:00] НАЗВАНИЕ КАНАЛА. [00:00:05] Тема: Введение в ML."
```

**Интерфейс:**

```python
@dataclass
class Video:
    transcription: str          # Полная транскрипция с таймкодами
    frame_descriptions: str     # Описания кадров с таймкодами
    frame_ocr: str              # OCR кадров с таймкодами
    duration: float | None
    
    def render(self) -> str:
        ...

class VideoProcessor:
    def __init__(self, photo_processor: PhotoProcessor, audio_processor: AudioProcessor,
                 frame_extraction_fps: float = 1.0, max_frames: int = 1000):
        ...
    
    def process(self, video_path: str | Path) -> Video:
        ...
```

### 6.5 PDFProcessor (существующий)

Используется `processor.complex_processor.pdf.PDFProcessor` как есть.

- Каждая страница рендерится в изображение (PyMuPDF, настраиваемый dpi) → `PhotoProcessor.run()`
- Лимит: `pdf.max_pages` (дефолт 1000). Если страниц больше — равномерная выборка, WARNING в лог
- Встроенные изображения **не** извлекаются отдельно (только рендер страниц)

### 6.6 Офисные процессоры (существующие)

Все используются из `processor.complex_processor.*` как есть:

| Формат | Процессор | Цепочка |
|--------|-----------|---------|
| DOCX/DOC | `WordProcessor` | LibreOffice → PDF → рендер страниц → PhotoProcessor |
| PPTX/PPT | `PowerPointProcessor` | LibreOffice → PDF → рендер слайдов → PhotoProcessor |
| XLSX | `XLSXProcessor` | openpyxl → CSV по листам |
| ODT | `ODTProcessor` | LibreOffice → PDF → PhotoProcessor |
| ODP | `ODPProcessor` | LibreOffice → PDF → PhotoProcessor |
| ODS | `ODSProcessor` | LibreOffice → XLSX → CSV |
| ODG | `ODGProcessor` | LibreOffice → PDF → PhotoProcessor |

Если LibreOffice не установлен — ошибка на healthcheck, система не запускается.

### 6.7 Инициализация процессоров

При старте BScrap все необходимые процессоры создаются **один раз** через их `build(config_path)` методы и переиспользуются для всех задач. Конфиг-файлы для `fast_ai` и `processor` указываются в основном `config.yaml` (см. п. 9).

## 7. Эмбеддинги

### Chunking

Библиотека: `langchain_text_splitters.RecursiveCharacterTextSplitter`

Параметры (из `config.yaml`):
- `chunk_size` (дефолт 512)
- `chunk_overlap` (дефолт 50)
- `separators` (дефолт `["\n\n", "\n", ". ", " "]`)

### Генерация эмбеддингов

Через `fast_ai.llm.FastLLM` — запрос к Ollama embedding API (`/api/embed`) или OpenAI embeddings API (`/v1/embeddings`), в зависимости от сконфигурированного провайдера.

Модель настраивается в `config.yaml` (`embedding_model`).

### Pipeline

1. `generated_text` → chunking → список чанков
2. Каждый чанк → запрос эмбеддинга → вектор
3. Вставка в таблицу `content_chunks`: `chunk_index`, `chunk_text`, `embedding`
4. Если `generated_text` пустой — эмбеддинги **не создаются**

### Размерность вектора

`VECTOR(N)` — размерность `N` берётся из `config.yaml` (`embedding_dimensions`). Используется только при создании/миграции таблицы. Runtime-валидация длины вектора не выполняется.

## 8. Модель данных (PostgreSQL + pgvector)

### Таблица `content_records`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | `gen_random_uuid()` |
| source_service | VARCHAR NOT NULL | Название сервиса (`telegram`, `youtube`, `email`, ...) |
| source_url | TEXT | Ссылка на первоисточник (nullable) |
| original_text | TEXT | Исходный текст (если передан в запросе) |
| generated_text | TEXT | Полный склеенный текст: обработанный входной текст + render() всех файлов |
| metadata | JSONB | Произвольные метаданные сервиса |
| created_at | TIMESTAMPTZ NOT NULL DEFAULT now() | Время создания записи |
| processed_at | TIMESTAMPTZ | Время завершения обработки |

**Constraints:**
- `UNIQUE (source_service, source_url) WHERE source_url IS NOT NULL` — partial unique index, проверка дубликатов только при заполненном `source_url`

### Таблица `content_chunks`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | `gen_random_uuid()` |
| content_record_id | UUID (FK) | → `content_records.id`, ON DELETE CASCADE |
| chunk_index | INTEGER NOT NULL | Порядковый номер чанка (начиная с 0) |
| chunk_text | TEXT NOT NULL | Текст чанка |
| embedding | VECTOR(N) | Вектор эмбеддинга, N из конфига |
| created_at | TIMESTAMPTZ NOT NULL DEFAULT now() | |

### Таблица `content_files`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | `gen_random_uuid()` |
| content_record_id | UUID (FK) | → `content_records.id`, ON DELETE CASCADE |
| original_filename | VARCHAR | Имя загруженного файла |
| file_type | VARCHAR | MIME-тип |
| file_size | BIGINT | Размер в байтах |
| extracted_text | TEXT | Текст, извлечённый из файла (`render()` процессора) |
| created_at | TIMESTAMPTZ NOT NULL DEFAULT now() | |

Файлы на диске **не сохраняются**. Полей `storage_path`, `is_stored` нет.

### Таблица `task_log`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | task_id |
| status | VARCHAR NOT NULL | `pending` / `processing` / `completed` / `failed` |
| source_service | VARCHAR | Сервис-источник |
| source_url | TEXT | Ссылка |
| content_record_id | UUID (FK, nullable) | → `content_records.id` (заполняется при completed) |
| error_message | TEXT | Описание ошибки (при failed) |
| created_at | TIMESTAMPTZ NOT NULL DEFAULT now() | Время создания задачи |
| started_at | TIMESTAMPTZ | Время начала обработки |
| completed_at | TIMESTAMPTZ | Время завершения |

Retention / автоочистка `task_log` не предусмотрена — таблица растёт неограниченно.

## 10. Запуск и остановка

### Запуск

```bash
python main.py
```

Последовательность:
1. Загрузка `config.yaml` + `.env`
2. Выполнение healthcheck (п. 3.4)
3. Если healthcheck не пройден → вывод ошибок в лог, **завершение с ненулевым кодом**
4. Инициализация БД-подключения (SQLAlchemy)
5. Инициализация SQLite-очереди
6. Сборка всех процессоров через `build()` (один раз)
7. Запуск воркера (polling очереди)
8. Запуск FastAPI (uvicorn)

### Остановка (SIGTERM / SIGINT)

Worker — встроенный, один. При получении сигнала:

1. Сервер перестаёт принимать новые запросы
2. Если воркер в данный момент обрабатывает задачу — **ждём завершения** текущей задачи (graceful)
3. Все задачи в очереди со статусом `processing` (не должно быть больше одной, но на случай сбоя) → статус откатывается в `pending`
4. Завершение процесса

## 11. Crash log

При ошибке обработки задачи (исключение в pipeline):

1. Данные задачи записываются в crash log
2. Формат: текстовый файл с JSON-строками (одна строка = одна задача)
3. Путь: `crash_log_path` из `config.yaml` (дефолт `./logs/crash.log`)
4. Программа **только пишет** в crash log, автоматического переигрывания нет
5. В `task_log` статус → `failed`, `error_message` заполняется текстом исключения

**Формат записи:**

```json
{"task_id": "uuid", "source_service": "telegram", "source_url": "...", "error": "traceback...", "timestamp": "2026-04-19T12:00:00Z", "payload": {...}}
```

## 12. Логирование

- **Консоль:** human-readable формат, уровень из `config.yaml` (`logging.level`)
- **Файл:** JSON-формат, один файл на запуск, имя = дата-время запуска (например `2026-04-19_14-30-00.log`)
- **Папка:** `logging.log_dir` (дефолт `./logs/`)
- **Содержимое каждой записи:** `timestamp`, `level`, `module`, `message`, `task_id` (если применимо), `traceback` (при ошибках)

Логируется:
- Старт/стоп системы
- Healthcheck результаты
- Каждый входящий запрос (`POST /process`)
- Старт/завершение обработки каждой задачи
- Каждый этап pipeline (какой процессор, для какого файла, сколько заняло)
- WARNING при уменьшении fps (видео) или при равномерной выборке страниц (PDF)
- WARNING при пропуске файла с неизвестным расширением
- ERROR при ошибках обработки

## 14. Временные файлы

- Входящие файлы сохраняются в `temp_dir` из `config.yaml` (дефолт `./temp/`)
- Каждая задача получает свою подпапку: `temp/<task_id>/`
- После обработки (успешной или нет) — подпапка задачи **удаляется целиком**
- Оригиналы файлов **не сохраняются** на диск — только временно на период обработки

---

## 15. Нефункциональные требования

- **Расширяемость:** новый формат = новый процессор + регистрация в `registry.py`. Если формат обрабатывается существующим процессором из `processor/` — достаточно добавить маппинг
- **Отказоустойчивость:** SQLite-очередь персистентная; при крашах задачи логируются, статус → `failed`; при рестарте задачи со статусом `processing` откатываются в `pending`
- **Наблюдаемость:** подробное логирование каждого этапа pipeline
- **Явность над неявностью:** система не запускается при невалидной конфигурации (нет CUDA, нет ffmpeg, нет LibreOffice, нет подключения к PostgreSQL/Ollama)
